## 上一篇的比方，还记得吗

我们说过：你派小王去办事。

- **Skill** 是你交代的**规矩**——「咱公司说『订单』都含税」。
- **MCP** 是你给的**工具**——钥匙、门禁卡、车。
- **Hook**（这篇的主题）是——**门口的安检**。

小王有了规矩、有了钥匙，但有些事你**绝不想让他干**——比如把公司数据库删了、把生产环境覆盖了。你不可能每次都跟在他后面盯着。**Hook 就是你设在门口的安检：他每次要做某个动作，先过这道检查，违规的直接拦下。**

## Hook 到底是什么

一句话：**Hook 是「在某个时机，自动跑你指定的一段脚本」。**

它不是给 Claude 的知识（那是 Skill），也不是给 Claude 的工具（那是 MCP）。它是**你给自己装的一道自动化关卡**——Claude 在执行某些操作前后，会自动触发你写好的脚本，你可以检查、记录、甚至拦下它。

举个最实用的例子：**Claude 每次要跑 bash 命令前，先过你的检查——如果命令是 `rm -rf` 这种危险操作，直接拦下，不让它执行。** 这就是本文要手写的 hook。

## 四件套全了：Hook vs Skill vs MCP vs Agent

三篇文章读下来，Claude Code 的四个核心概念凑齐了：

| | 是什么 | 谁定的 | 例子 |
|---|---|---|---|
| **Skill** | 知识 / 纪律 | 你写给 Claude 读的 | 「这个项目的术语」「审查清单」 |
| **MCP** | 工具 / 能力 | 你给 Claude 接的 | 「能读文件」「能查数据库」 |
| **Hook** | 自动关卡 | 你给自己装的 | 「跑命令前先检查」「写完自动跑测试」 |
| **Agent** | 一个分身 | 你或主 agent 派的 | 另开一个 Claude 去独立干活 |

记住这组分工：
- Skill 让它**知道怎么想**
- MCP 让它**能动手去够**
- **Hook 让它动手时**不出格
- Agent 是**另派一个人**

## 光说不练假把式——手写一个

我们写一个**最小有用的 hook**：拦住 `rm -rf` 这种危险命令。写完你就懂了——hook 没什么神秘，就是「配置 + 一段脚本」。

> ⚠️ **安全说明**：这个 hook 是给你自己用的保护，只拦 Claude 想跑的危险命令，不影响你手动在终端敲命令。

### 第 1 步：写 hook 脚本

在你的项目里建一个文件 `.claude/hooks/block-rm-rf.sh`：

```bash
#!/bin/bash
# .claude/hooks/block-rm-rf.sh
# Claude 每次要跑 bash 命令前，会自动调用这个脚本，把命令喂给它。
# 我们检查命令里有没有 rm -rf，有就拦下。

COMMAND=$(jq -r '.tool_input.command')

if echo "$COMMAND" | grep -q 'rm -rf'; then
  echo "Blocked: rm -rf 太危险，已被 hook 拦下。" >&2
  exit 2   # exit 2 = 拦截，stderr 的内容会反馈给 Claude
fi

exit 0     # exit 0 = 放行，走正常流程
```

读这段脚本在干什么：
- `jq -r '.tool_input.command'` —— Claude 把要执行的命令以 JSON 形式喂进来，这行把命令提取出来（`jq` 是命令行 JSON 工具）
- `grep -q 'rm -rf'` —— 检查命令里有没有危险字符串
- `exit 2` —— **这是关键**：退出码 2 表示「拦截」，而且 `>&2` 输出到 stderr 的那句话，会变成给 Claude 的反馈（它会知道为什么被拦）
- `exit 0` —— 退出码 0 表示「不管，放行」

### 第 2 步：把脚本「挂」到 PreToolUse 时机

脚本写好了，还要告诉 Claude Code「**什么时候**跑它」。这就是 hook 的另一半——**时机**。

在你的项目根目录建 `.claude/settings.json`（如果已有就编辑），加入：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/block-rm-rf.sh"
          }
        ]
      }
    ]
  }
}
```

读这段配置在干什么：
- `PreToolUse` —— **时机**：在 Claude「使用工具之前」触发（还有 `PostToolUse` 等其他时机）
- `matcher: "Bash"` —— 只匹配 Bash 工具（Claude 要跑终端命令时才检查，写文件、读文件不管）
- `command` —— 要跑的那段脚本（就是第 1 步写的）

**合起来**：Claude 每次准备跑 bash 命令前，自动执行你的 `block-rm-rf.sh`，把命令交给它检查。

### 第 3 步：重启 Claude Code，让配置生效

和 MCP 一样，hook 配置在启动时加载。**退出、重开**。

## 🎉 见证时刻

重启后，让 Claude 跑一条危险命令——比如跟它说：

> 「帮我清一下 `/tmp/build` 目录」

如果它想用 `rm -rf /tmp/build`，**会被你的 hook 拦下**，你会看到类似：

```
Blocked: rm -rf 太危险，已被 hook 拦下。
```

Claude 收到这个反馈后，会换个安全的做法（比如用 `rm -r` 不带 `-f`，或问你怎么处理）。

**就是这个时刻。** 你写了一段十几行的脚本 + 一段配置，从此 Claude 在这个项目里**再也跑不了 `rm -rf`**——不是你每次盯着，是**自动**拦的。

## Hook 和 Skill 的关键区别（容易混）

新手常把 hook 和 skill 搞混。一句话区分：

- **Skill 是给 Claude 的**——你写给**它**读的知识，它读了「知道怎么做」
- **Hook 是给你自己的**——你设在它**必经之路上**的关卡，它过路时被检查

Skill 是建议（Claude 可以不遵循），**Hook 是强制**（exit 2 直接拦，绕不过）。所以**安全规则该用 hook，不该只写进 skill**——你想让它万无一失的事，别靠它自觉，装道关卡。

## 后续：还能干什么

拦 `rm -rf` 只是开始。Hook 的真正威力在于「**把纪律自动化**」：

- **PreToolUse + Bash** —— 拦危险命令（本文做的）
- **PostToolUse + Edit** —— 每次改完文件，自动跑格式化 / 类型检查
- **Stop** —— Claude 任务完成时，自动提醒「跑测试了吗」「提交了吗」
- **SessionStart** —— 每次开会话，自动加载项目上下文

凡是「**每次都该做、但总忘**」的事，都可以挂成 hook——让机器替你记住。

## 收尾

你刚做了三件事：写了段脚本、写了段配置、给 Claude 装了一道自动护栏。这就是 hook 的全部——**时机 + 脚本，把你想强制的规则自动化执行**。

回头看那个比方：Skill 是交代的规矩，MCP 是给的钥匙，**Hook 是门口的安检**。规矩是建议，安检是强制——你想万无一失的事，靠安检，别靠自觉。
