## 你大概遇到过这种事

你跟 Claude 说「帮我审一下这段代码」，它审了，但审得很泛——安全性、命名、性能，什么都提一点，没重点。

下一次你又让它审，它又换了个角度。每次都得你重新交代「我关心的是安全」「我关心的是性能」。

**问题不在于 Claude 不聪明，而在于它每次都是「空手」上阵**——它不知道你这个项目、这次任务，到底该按什么标准来。

Skill 就是为解决这个而生的。

## Skill 到底是什么

一句话：**Skill 是一份带说明书的 markdown 文件，Claude 会在「该用它的时候」自动读它，然后按里面的规矩做事。**

它不是一个程序，不是一段代码，不是什么黑魔法。就是一个 markdown 文件，叫 `SKILL.md`，长这样：

```markdown
---
name: review-for-security
description: 审查代码的安全漏洞、认证缺陷、注入风险。当代码涉及认证、用户输入、数据库查询时使用。
---

## 审查清单

逐条检查：

1. 有没有硬编码的密钥、密码、token
2. 用户输入有没有经过转义（防 XSS、SQL 注入）
3. 认证逻辑有没有绕过的可能
4. 文件路径有没有被用户控制（防路径穿越）
```

注意 frontmatter 里那个 `description`——**它是整个 Skill 的灵魂**。

## description 不是给人看的，是给 Claude 看的

这是新手最容易误解的一点。

你以为 `description` 是文档、是给人读的解释？**不是。它是给 Claude 看的触发条件**——要把需要触发的关键词和场景尽量写清楚。写错了，这个 skill 要么不被调用，要么不被正确调用。

Claude 每次接到你的任务，会扫一眼所有 skill 的 `description`，判断哪个和当前任务相关。如果你的 description 写得含糊（比如「审查代码」），Claude 就不知道什么时候该用它；写得精准（比如「当代码涉及认证、用户输入、数据库查询时使用」），它就能在对的时刻自动触发。

## Skill vs Slash 命令 vs MCP vs Agent

这四个词初学者最容易混。一张表说清：

| | 是什么 | 谁触发 | 干什么 |
|---|---|---|---|
| **Skill** | 带说明书的 markdown | 默认 Claude 自动触发（也可手动） | 注入**知识和纪律**（「该按什么标准做」） |
| **Slash 命令** | `.claude/commands/` 里的 markdown | **你**敲 `/名字` | 一个固定的**操作入口**（「执行这套流程」） |
| **MCP** | 连接外部服务器的协议 | Claude 调用 | 给 Claude 接**外部工具和数据**（「能调用的能力」） |
| **Agent（子 agent）** | 独立的执行体 | 你或主 agent 分派 | 独立完成一个**子任务**（「另一个干活的」） |

记一组对比就懂了：

- Skill 让 Claude **知道该怎么想**（知识 / 纪律）
- MCP 让 Claude **能动手去够**（外部工具 / 数据）
- Slash 命令是**你按下的启动键**
- Agent 是**另开一个分身去干活**

打个比方：你派小王去办事。

- **Skill** 是出发前你交代的规矩——「咱公司说『订单』都含税」「签字别跳过张总」。这是**知识和纪律**。
- **MCP** 是你塞进他手里的工具——一把钥匙、一张门禁卡、一辆车。小王光有你的嘱咐办不成事，得有这些才**够得着**原本够不着的地方（数据库、文件、外部系统）。这是**能力和工具**。

一句话：Skill 让他知道**该怎么想**，MCP 让他**能动手去够**。

## 模型部分小结

- Skill = 带说明书的 markdown（`SKILL.md`）
- `description` 是给 **Claude** 看的触发判据，不是给人看的文档
- 它注入的是**知识和纪律**，和给工具的 MCP、给入口的 Slash、派分身的 Agent 是不同维度的东西

接下来动手——**5 分钟，手写一个真能用的 Skill**。

## 实操：手写你的第一个 Skill

我们做一个最常见、最实用的小 Skill：**项目术语词典**。

每个项目都有自己的黑话——「我们说的『订单』其实是含税的」「『活跃用户』指 30 天内登录过的」。你每次都要跟 Claude 解释一遍？不用。写进一个 Skill，Claude 以后在这个项目里就自动说你的语言。

### 第 1 步：建文件

在你的项目根目录下，建这个路径：

```
.claude/skills/project-glossary/SKILL.md
```

> 路径规则：`.claude/skills/<skill 名字>/SKILL.md`。文件夹名就是 skill 的名字。

### 第 2 步：写 frontmatter

```markdown
---
name: project-glossary
description: 本项目的术语词典。当讨论涉及 Task / Project / Done 这些概念时使用，确保用词与项目定义一致。
---
```

记住上面说的——`description` 写清楚「**什么时候用**」，Claude 才知道何时调它。

### 第 3 步：写术语（关键：让它们互相引用）

这是最有用的一招——**术语的定义里可以引用别的术语**。这样 Claude 读完，脑子里就有了一张关系网：

```markdown
## 术语

**Task（任务）**：
一件待办的事，属于某个 **Project**。每个 Task 有明确的状态（未开始 / 进行中 / **Done**）。

**Project（项目）**：
一组 **Task** 的集合，共享同一个目标。

**Done（已完成）**：
一个状态为「已完成」的 **Task**。判断标准是：通过验收且合并到主干。
```

注意到了吗？`Task` 的定义里用到了 `Project`，`Done` 的定义里用到了 `Task`。这种**交叉引用**让术语之间织成一张网，而不是一堆孤立的词条。

### 第 4 步：验证 Claude 真的读懂了

保存文件，回到 Claude Code，随便问一句：

> 「我现在手上的 **Task** 哪些还没 **Done**？按 **Project** 分组列给我。」

如果 Claude 能用 **Project / Task / Done** 这套词精准回答（而不是泛泛地说「你还有几个任务没做」），说明这个 Skill 已经生效了——它读了你写的术语，正在用你的语言跟你对话。

## 你刚才其实做了一件大事

回头看——你没装什么框架，没跑什么安装命令，就建了一个 markdown 文件。但 Claude 从此在这个项目里，会自动按你的术语、你的标准来对话。

这就是 Skill 的本质：**它把「你脑子里关于这个项目的知识」，变成了一份 Claude 会自动读取的说明书。**
